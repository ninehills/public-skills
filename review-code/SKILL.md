# 代码审查技能（review-code）

基于 Google 工程实践（[eng-practices](https://github.com/google/eng-practices)）整理的代码审查完整指南。当需要审查代码变更、PR、或审计现有代码时加载此技能。

## 核心原则

**代码审查的首要目的：确保代码库的整体代码健康状况随时间推移而改善。**

评审者应在 CL 明确改善系统代码健康状态时予以通过，即使 CL 并不完美。不存在"完美"代码，只有**更好**的代码。追求持续改善，而非追求完美。

### 三大原则

1. **技术事实和数据压倒意见和个人偏好。**
2. **风格问题上，风格指南是绝对权威。** 任何不在风格指南中的纯风格问题属于个人偏好，应保持与现有代码一致。如果没有既有风格，接受作者的写法。
3. **软件设计几乎从来不是纯风格问题或个人偏好。** 它们基于底层原则，应根据原则衡量，而非仅凭个人意见。如果作者能证明（通过数据或可靠工程原则）多种方案同样有效，评审者应接受作者的偏好。否则由标准软件设计原则决定。

## 一、审查总览

代码审查应关注以下维度：

- **设计**：代码设计是否合理、与系统适配？
- **功能**：代码行为是否符合作者意图？对用户是否有益？
- **复杂度**：能否更简单？未来其他开发者能否轻松理解和修改？
- **测试**：是否有正确、设计良好的自动化测试？
- **命名**：变量、类、方法等命名是否清晰？
- **注释**：注释是否清晰有用？注释应解释**为什么**而非**是什么**。
- **风格**：是否遵循风格指南？
- **文档**：是否同步更新了相关文档？
- **每一行**：审查被分配到的**每一行**代码。数据文件、生成代码可以略过，但人工编写的类/函数/代码块不能假定其内部没问题。
- **上下文**：在更广阔上下文中审视 CL——整个文件、整个系统。不接受降低代码健康状态的 CL。

## 二、审查流程

### 三步法浏览 CL

**第一步：整体视角**
- 阅读 CL 描述，理解变更做什么、为什么做。
- 这个变更本身是否合理？如果根本不应该做，立即回复说明原因并建议替代方案。
- 拒绝时要礼貌——"看起来你做了不错的工作，但我们正在移除 FooWidget 系统，不建议再做修改。是否考虑重构新的 BarWidget 类？"

**第二步：审查核心部分**
- 找到变更量最大的"主"文件，先审查这些核心部分。
- 如果发现重大设计问题，立即发送评论——不等其他部分审查完。因为如果设计问题严重，大量后续代码可能被废弃。
- 立即指出设计问题有两个原因：开发者通常在等待 review 时已开始基于此 CL 的后续工作；设计变更比小改动耗时更长。

**第三步：按合理顺序审查其余部分**
- 确认无重大设计问题后，按逻辑顺序浏览剩余文件。
- 常规做法是按代码审查工具展示的文件顺序逐一审查。
- 有时先读测试代码再读主代码有助于理解变更意图。

### 审查速度

- **关键指标是响应时间**，而非整个流程完成时间。快速响应比快速完成更重要。
- **一个工作日**是响应代码审查请求的最长时间。
- 如果你正在专注编码中，不要打断自己——等待工作中的断点再响应。
- 如果你太忙无法完整审查，可以：快速回复告知何时能处理、建议其他能更快响应的评审者、先给出一些整体性评论。
- **LGTM With Comments（带评论通过）**——在以下情况即使有未解决的评论也可以批准：
  - 你确信开发者会妥善处理剩余评论。
  - 评论不是必须由开发者处理的（如 Nit 类型）。
  - 建议是次要的（如排序 import、修复附近拼写错误）。
- **大 CL 策略**：要求开发者拆分为多个小 CL。如果无法拆分，至少写一些整体设计层面的评论。

### 跨时区审查

尽量在作者工作时间结束前回复，让他们能在下班前处理。如果作者已下班，确保在次日他们开始工作前完成审查。

## 三、审查标准详解

### 设计审查

最重要的审查维度。检查：
- CL 中各代码模块的交互是否合理？
- 变更应该放在当前代码库还是抽到库中？
- 是否与系统其余部分良好集成？
- 现在是添加此功能的合适时机吗？

### 功能审查

- CL 是否实现了开发者意图？开发者意图对用户是否有利？
- 思考边界情况、并发问题、站在用户角度思考。
- 特别关注：**UI 变更**（难以仅靠读代码判断影响，可要求开发者演示）、**并行编程**（可能导致死锁或竞态条件，必须仔细推理）。
- 可以通过打补丁并亲自尝试来验证 CL。

### 复杂度审查

- 逐级检查：单行是否过于复杂？函数是否过于复杂？类是否过于复杂？
- "过于复杂"的含义：**代码读者无法快速理解**、**开发者调用或修改时容易引入 bug**。
- 特别警惕**过度工程**：代码做得比需要的更通用，或添加了当前系统不需要的功能。鼓励开发者解决**现在**已知需要解决的问题，而非猜测未来可能需要解决的问题。
- 未来问题应在实际到来时再解决，届时你才能看到它在现实中的真正形态和需求。

### 测试审查

- 要求与变更匹配的单元测试、集成测试或端到端测试。
- 测试应和产品代码在同一个 CL 中（紧急情况除外）。
- 确保测试正确、合理、有用：
  - 代码损坏时测试会真的失败吗？
  - 底层代码变更时测试会误报吗？
  - 断言是否简单有用？
  - 不同测试方法之间是否合理分离？
- 记住：测试也是需要维护的代码。不要因为测试不在主二进制中就接受测试中的复杂度。

### 命名审查

- 好名字足够长以完全传达该项是什么/做什么，但不至于长到难以阅读。

### 注释审查

- 注释是否清晰、用可理解的语言？
- 注释通常应解释**为什么**而非**是什么**。如果代码本身不够清晰到自我解释，应该简化代码而非加注释。
- 例外：正则表达式和复杂算法通常需要通过注释来解释它们在做什么。
- 查看 CL 之前的注释：是否有可以移除的 TODO、是否有反对此变更的旧注释等。
- 注释不同于**文档**（类/模块/函数的文档应表达目的、用法和行为）。

### 风格审查

- 确保 CL 遵循相应风格指南。
- 如果想改进风格指南之外的内容，用 "Nit:" 前缀标注，让开发者知道这不是强制性的。
- 不要仅基于个人风格偏好阻止提交。
- CL 不应将大规模风格变更与功能变更混在一起——格式化和功能应分开提交。

### 一致性

- 如果现有代码与风格指南不一致：风格指南是绝对权威，CL 应遵循指南。
- 如果风格指南是建议而非要求：倾向于遵循风格指南，除非局部不一致会造成更大混淆。
- 无论哪种情况，鼓励作者提交 bug 并添加 TODO 来清理现有代码。

### 文档审查

- 如果 CL 改变了用户构建、测试、交互或发布代码的方式，检查是否同步更新了 README、文档页面和生成的参考文档。
- 如果 CL 删除或废弃代码，考虑文档是否也应删除。

## 四、代码审查评论写法

### 四项要点

1. **友善**
2. **解释原因**——让开发者理解你提出评论的意图、遵循的最佳实践、或你的建议如何改善代码健康。
3. **平衡直接指导和指出问题让开发者自己决定**——指出问题让开发者决策有助于他们学习和成长。但有时直接指令、建议甚至代码更有帮助。
4. **鼓励开发者简化代码或添加代码注释**，而非仅仅向你解释复杂性。

### 礼貌原则

- 总是评论**代码**，永远不要评论**开发者**。

```
差: "你为什么在这里用线程？明显从并发中得不到任何好处。"
好: "这里的并发模型给系统增加了复杂度，但我看不到实际的性能收益。因为没有性能收益，最好用单线程而不是多线程。"
```

### 评论严重程度标签

- **Nit**：小事，技术上应该做但影响不大。
- **Optional / Consider**：可能是个好主意，但不严格要求。
- **FYI**：不期望在本 CL 中修改，但未来可以考虑。

使用标签让审查意图明确，帮助作者确定优先级，避免误解。

### 关于解释

如果你要求开发者解释一段代码，通常结果应该是他们**重写代码使其更清晰**。偶尔添加代码注释也是合适的回应（前提是不只是解释过于复杂的代码）。

仅在代码审查工具中写的解释对未来代码读者没有帮助。只在少数情况下可接受——例如你审查的是不熟悉的领域，开发者解释的内容普通代码读者本来就知道。

## 五、处理反对意见

### 谁是对的？

- 当开发者不同意你的建议时，先思考他们是否正确——他们通常比你更接近代码。
- 如果开发者的论点合理且在代码健康角度说得通，告诉他们是对的，让问题过去。
- 但如果开发者不对，进一步解释为什么你认为你的建议正确。好的解释展示你理解开发者的回复，并提供关于为什么要求变更的额外信息。
- 改善代码健康往往发生在小步骤中。可能需要几轮解释才能让建议真正被理解。
- 始终保持礼貌，让开发者知道你**倾听**了他们的说法，只是你**不同意**。

### 关于"以后再清理"

- 开发者常表示会在后续 CL 中清理——经验表明，除非开发者在当前 CL 之后**立即**写清理 CL，否则很少真正发生。
- 不是开发者不负责任，而是他们有很多工作要做，清理在大量工作中被遗忘。
- 通常最好坚持开发者在代码进入代码库之前**现在**清理。让"以后再清理"是代码库退化的常见方式。
- 如果 CL 引入了新的复杂度，必须在提交前清理（除非紧急情况）。
- 如果 CL 暴露了周围问题且现在无法解决，开发者应提交 bug 并分配给自己，可选在代码中写 TODO 引用该 bug。

### 关于严格性的抱怨

- 如果你从宽松的审查转向严格，一些开发者会抱怨。提高审查速度通常能让这些抱怨消失。
- 可能需要几个月抱怨才消失，但当开发者看到严格审查产生的优秀代码时，甚至最强烈的反对者也可能变成最坚定的支持者。

### 冲突解决

如果无法解决冲突，升级途径：更广泛的团队讨论 → TL 介入 → 代码维护者决策 → EM 介入。**不要让 CL 因为作者和评审者无法达成一致而搁置。**

## 六、开发者指南

### 写好 CL 描述

CL 描述是变更的永久公开记录，应传达：

1. **做了什么变更？** 摘要应让读者无需阅读整个 CL 即可感知变更内容。
2. **为什么做这些变更？** 作为作者的上下文、做出的决策、源代码中未反映的信息。

**第一行规则：**
- 简短摘要具体做了什么。
- 完整句子，祈使句（如 "删除 FizzBuzz RPC 并替换为新系统"）。
- 第一行后空一行。
- 第一行应独立存在，让未来搜索代码历史的人能快速理解。

**描述正文：**
- 补充细节，包括解决的问题、为什么这是最佳方案、方案的不足之处。
- 包括背景信息：bug 编号、基准测试结果、设计文档链接。

**坏的描述示例：** "Fix bug." / "Fix build." / "Add patch." / "Phase 1."

**好的描述示例：**
```
RPC: 移除 RPC 服务器消息空闲列表的大小限制。

像 FizzBuzz 这样的服务器有非常大的消息，能从复用中受益。
增大空闲列表，并添加一个 goroutine 随时间缓慢释放空闲列表条目，
使空闲服务器最终释放所有空闲列表条目。
```

### 写小 CL

**小 CL 的好处：** 审查更快、审查更彻底、更少引入 bug、被拒绝时浪费更少工作、更容易合并、更容易设计好、减少等待审查的阻塞、更容易回滚。

**多大算小？** 一般来说，一个 CL 应是一个**自包含的变更**——只解决一件事。通常 100 行是合理的，1000 行通常太大。文件的分散程度也影响"大小"——一个文件中的 200 行可能 OK，但散在 50 个文件中通常太大。

**拆分策略：**
- **基于堆叠**：写完一个小 CL 发出审查后，立即基于此 CL 写下一个。
- **按文件拆分**：需要不同评审者的文件分组，各自自包含。
- **水平拆分**：在技术栈各层之间创建共享代码或桩，隔离变更。如先发 proto 定义 CL，再发使用它的代码 CL。
- **垂直拆分**：按功能拆分，每个功能是独立的全栈实现轨迹。
- **重构分离**：重构应放在独立 CL 中，不与功能变更或 bug 修复混在一起。
- **测试代码在同一个 CL**：逻辑变更应包含新的或更新的测试。

### 处理评审者的评论

- **不要当做个人攻击**——审查的目标是维护代码库和产品质量。
- **永远不要愤怒回复审查评论**——这是严重违反职业礼仪的行为。
- **首先修复代码**——如果评审者不理解某段代码，首先澄清代码本身；如果无法澄清，添加代码注释；只有当注释也没意义时才在审查工具中解释。
- **协作思维而非对抗**——如果不同意评论：
  ```
  差: "不，我不打算这么做。"
  好: "我选择 X 是因为 [这些利弊/权衡]。我的理解是用 Y 会更差因为 [这些原因]。
       你是建议 Y 更好地服务于原始权衡，还是我们应该以不同方式权衡，还是其他？"
  ```
- 如果你知道评审者不知道的上下文，提供更多上下文。通常可以基于技术事实达成共识。

## 七、紧急情况

### 什么是紧急情况？

**小**变更，允许：重大发布继续进行而非回滚、修复严重影响生产用户的 bug、处理紧迫的法律问题、关闭重大安全漏洞等。

只有在紧急情况下，评审者才应将审查速度和代码正确性（是否真正解决紧急问题）置于其他一切之上。

紧急情况解决后，应重新审查紧急 CL 并做更彻底的审查。

### 什么不是紧急情况？

- 想本周发布而非下周
- 开发者花很长时间开发了功能，非常想让 CL 通过
- 评审者都在另一时区/外出
- 周五下班前想把 CL 提交
- 经理说今天必须完成审查
- 回滚导致测试失败或构建失败的 CL

### 什么是硬截止日期？

**错过会导致灾难性后果**的截止日期才是硬截止日期。如合同义务、产品完全失败、硬件制造商一年只发布一次等。

发布延迟一周不是灾难。大多数截止日期是软截止。

如果团队经常在周期末尾以肤浅审查提交"必须通过"的 CL，应修改流程，让大型功能变更在周期早期完成并有足够时间接受充分审查。

## 八、AI 审查执行指南

作为 AI，执行代码审查时遵循以下流程：

### 审查准备

```bash
# 获取 diff
git diff --cached
# 如果为空，尝试
git diff HEAD~1 HEAD
```

### 审查输出格式

```markdown
## 审查总结
[1-2 句整体评估]

## 设计层面
- [设计相关评论]

## 关键问题（必须修复）
- 问题 1：[描述 + 建议修复方案]
- 问题 2：...

## 建议（可选）
- 建议 1：[描述]

## 好的实践（肯定）
- [开发者做得好的地方]

## 问题
- [需要开发者澄清的问题]
```

### 常见问题速查

**Python**
```python
# 差: SQL 注入风险
cursor.execute(f"SELECT * FROM users WHERE id = {user_id}")
# 好: 参数化查询
cursor.execute("SELECT * FROM users WHERE id = ?", (user_id,))

# 差: Shell 注入
os.system(f"ls {user_input}")
# 好: 安全子进程
subprocess.run(["ls", user_input], check=True)

# 差: 危险 eval/exec
eval(user_input)
# 好: 使用安全的解析方式
```

**JavaScript**
```javascript
// 差: XSS 风险
element.innerHTML = userInput;
// 好: 安全文本内容
element.textContent = userInput;
```

### 需要特别关注的信号

- 硬编码的密钥、API Key、凭证
- 缺少输入验证
- SQL 查询使用字符串拼接
- 文件操作未验证路径（路径穿越风险）
- 外部调用缺少错误处理
- 遗留的调试 print/console.log
- 被注释掉的代码
- 新代码没有对应测试
- 过度工程（做了当前不需要的抽象/功能）
- 并发代码没有正确处理竞态条件
- 重构和功能变更混在同一个 CL 中
- 大规模格式化变更和逻辑变更混在一起
